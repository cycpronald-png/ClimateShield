  <action>
The TypeScript compiler reports 3 errors in RiskScoreGauge.tsx, all caused by `composite_risk_score` being typed as `number | undefined` in the `WeatherReading` interface. The Phase 6 gray-badge changes introduced these by using `composite_risk_score` directly without type narrowing.

Fix approach — use explicit type narrowing to extract the score into a properly-typed variable:

**In the `AllStationsGrid` function (around line 163-167):**
Current code:
```typescript
const hasPersistedScore = r.composite_risk_score != null;
const score = hasPersistedScore ? r.composite_risk_score : 0;
```
Change to:
```typescript
const hasPersistedScore = r.composite_risk_score != null;
const score: number = hasPersistedScore ? r.composite_risk_score! : 0;
```
The `!` non-null assertion is safe here because `hasPersistedScore` guarantees `composite_risk_score` is not null. Alternatively, use:
```typescript
const persistedScore = r.composite_risk_score;
const hasPersistedScore = persistedScore != null;
const score: number = hasPersistedScore ? persistedScore : 0;
```
This second approach is preferred — it captures `persistedScore` in a local const so TypeScript can narrow it through the `!= null` check. The `score` variable will then be correctly typed as `number`.

**Same pattern in the `SingleGauge` function (around line 50):**
The SingleGauge already uses:
```typescript
const hasPersistedScore = reading.composite_risk_score != null;
const score = liveScore?.value ?? reading.composite_risk_score ?? 0;
```
This should work if `liveScore?.value` is `number | undefined` and the nullish coalescing chain resolves to `number`. But TypeScript may not narrow `reading.composite_risk_score` properly. Fix to:
```typescript
const persistedScore = reading.composite_risk_score;
const hasPersistedScore = persistedScore != null;
const score: number = liveScore?.value ?? persistedScore ?? 0;
```

**Also check line 212 area** — `score.toFixed(0)` where `score` is `number | undefined`. This is inside the `hasPersistedScore && (...)` guard block, so it's logically safe but TypeScript doesn't narrow through the boolean guard. After the fix above, `score` will be `number` and this error resolves.

**Do NOT change the `WeatherReading` interface** — `composite_risk_score?: number` correctly represents that the API may return readings without a score. The fix must be in the component's type narrowing, not in making the type less precise.
